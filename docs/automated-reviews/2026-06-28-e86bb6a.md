## Architecture candidates

### Candidate 1 - Strong

- Files: `extensions/task/task.ts`, `extensions/task/task-state.ts`, `extensions/task/task-run-reviewer.ts`, `extensions/task/task-worker.ts`, `extensions/task/task-verify.ts`, `tests/task-extension.test.ts`
- Module: task run lifecycle and review handoff
- Problem: one Module now owns run orchestration, abort semantics, run-history writes, reviewer progress streaming, review persistence, and the “let reviewer fix it” edit handoff. Its Interface is broad enough that callers and tests must understand worker execution, verify semantics, UI widget refresh, and session-state persistence together.
- Solution: deepen one Module around “task run review state” only. Let it own the run record, the lightweight last-run reference, and the reviewer/edit handoff, while `/task` command flow remains the adapter.
- Benefits: better Locality for future run/review bugs, more Leverage from a smaller test surface, and an obvious Seam where session persistence can store a reference instead of a full report blob. The deletion test passes: if this Module disappeared, the same complexity would reappear across `/task run`, `/task review-last-run`, and session restore.

### Candidate 2 - Worth exploring

- Files: `extensions/ui-brand.ts`, `extensions/ui-brand-utils.ts`, `tests/ui-brand-extension.test.ts`, `tests/ui-titlebar.test.ts`
- Module: branded UI session snapshot
- Problem: title ownership, header coloring, footer usage sampling, stale-context tolerance, and active-session cwd lookup still live in one Implementation. The Interface is wider than it needs to be because any change to one branded surface requires understanding all of them.
- Solution: concentrate the current branded session snapshot into one deeper Module that materializes cwd, title state, context usage, and logo tone data once per render tick, then let header/footer/title adapters consume that snapshot.
- Benefits: more Locality when session replacement or title ownership changes, and more Leverage from render-focused tests instead of broad extension hook tests.

### Candidate 3 - Worth exploring

- Files: `skills/task-creator/SKILL.md`, `extensions/task/task-worker.ts`, `extensions/task/task-verify.ts`, `extensions/task/task-book.ts`
- Module: taskbook authoring contract
- Problem: the authoring Interface now spans a long prose skill plus runtime conventions such as `TASK_DIR`, `TASK_INPUT`, `TASK_OUTPUT_DIR`, and `requiredTools`. The same contract is explained in several places, so drift risk rises even though the underlying behavior is small.
- Solution: keep one canonical authoring Module or sample/template for taskbook layout and runtime env rules, and have `task-creator` point to that source instead of restating the full contract in prose.
- Benefits: stronger Locality for future taskbook format changes and more Leverage from changing one source of truth instead of code plus skill prose plus examples.

## Top recommendation

Explore Candidate 1 first. This diff made the task run lifecycle deeper in behavior but not deeper in structure: the same file now owns more run/review state while also persisting large review payloads into session history. A small, explicit run-review Module is the clearest path to recover Locality without reopening the whole task system.
