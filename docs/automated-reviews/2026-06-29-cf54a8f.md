## Architecture candidates

### Candidate 1 - Strong

- Files: `extensions/shared/settings-io.ts`, `extensions/shared/language.ts`, `extensions/shared/ui-language.ts`, `tests/language.test.ts`, `tests/ui-language.test.ts`
- Module: settings persistence
- Problem: settings Module 已经负责 BOM-safe read/write，但 key 删除的 Implementation 泄漏到两个调用 Module。Interface 缺少“删除一个 settings key”这一行为，导致 Locality 变差。
- Solution: deepen settings Module：一个 Interface 同时覆盖 read、update、delete key；`language` 和 `ui-language` 只表达各自的 key 和默认语义。
- Benefits: Locality: settings 文件格式和 BOM 处理只在一处；Leverage: 后续新增持久偏好不用复制写文件代码；test surface 也集中在 settings Interface。

### Candidate 2 - Worth exploring

- Files: `extensions/shared/ui-language.ts`, `extensions/index.ts`, `extensions/ui-brand.ts`, `extensions/ui-brand-utils.ts`, `extensions/chrome-cdp/index.ts`, `extensions/mcp/commands.ts`, `extensions/plan-mode.ts`
- Module: UI copy lookup
- Problem: `uiText()` 的 Interface 要求每个 caller 同时知道中文、英文和部分翻译 fallback；新增语言时 Implementation 集中在 `TRANSLATIONS`，但调用端仍携带大量 copy 细节。
- Solution: 仅当 UI 文案继续增长时，把常用菜单/状态文案深化成 keyed copy Module；caller 传 key 和变量，翻译表吸收语言选择和 fallback。
- Benefits: Locality: copy 变化少碰命令实现；Leverage: 多语言覆盖从 N 个 caller 收敛到一个 Interface。当前先别做，避免为了 i18n 造大框架。

### Candidate 3 - Speculative

- Files: `extensions/shared/worker-lifecycle.ts`, `extensions/chrome-cdp/tab-session.ts`, `extensions/chrome-cdp/index.ts`, `extensions/task/task-worker.ts`, `extensions/subagent.ts`
- Module: worker resource lifecycle
- Problem: 这是一个 one-adapter seam：目前只有 CDP tab lifecycle 一个 Adapter。按删除测试看，删掉 seam 会让 `task-worker` 重新知道 CDP 细节，所以它仍在赚 Locality。
- Solution: 保持现状。只有当第二种 worker 资源出现时，才深化 Interface；否则不要扩展成 registry、plugin lifecycle 或多 Adapter 管理器。
- Benefits: Locality 已够用；继续加抽象会降低 Depth。

## Top recommendation
