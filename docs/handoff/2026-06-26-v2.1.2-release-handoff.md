## 二、这是什么项目

**ugk-core** 是基于 [pi](https://github.com/earendil-works/pi)(pi-coding-agent)定制的编码 agent **ugk-pi-agent** 的核心代码仓。它不是一个独立产品,而是把 pi 扩展成一套带品牌 UI、任务委托、实时监督、定时任务、本地 Chrome/MCP 控制等能力的 agent 工具链。

**一句话定位**:让用户在终端里用 `/task`、`/judge`、`/cron` 等 slash 命令驱动 agent 干活,而非手动编排。

**必读入口**(按顺序):
1. **`AGENTS.md`**(仓库根) - agent 运行时上下文,描述已实现能力、slash 命令、权限门、关键约定。**改任何运行时行为前先读它。**
2. **`docs/DEVELOPMENT.md`** - 开发侧约定(AGENTS.md 里明确说"不在运行时注入,见此文件")。改 ugk-core 代码前必读。
3. **`README.md`** - 面向用户的使用说明。

**仓库结构**:
```text
extensions/      <- 核心:所有 agent 能力的扩展代码(重点)
  task/          <- /task 固定任务委托系统(本阶段主战场)
  judge/         <- /judge 实时监督模式
  chrome-cdp/    <- 本地登录态 Chrome 控制
  mcp/           <- MCP server 接入
  doctor/        <- 自检
  shared/        <- 共享工具
  index.ts       <- 入口:危险 bash 门、扩展注册
  subagent.ts    <- 子进程 spawn(worker 走这个)
  ...
agents/          <- agent 配置(worker/checker/guide/reviewer 等)
prompts/         <- agent prompt
skills/          <- 系统自带 skill(随包走)
user-skills/     <- 用户 skill(手动安装)
themes/          <- UI 主题
tests/           <- 测试(node:test + TS)
docs/            <- 设计文档/handoff/报告(历史丰富)
bin/             <- CLI 入口(ugk.js)
cron/            <- 定时服务
```

## 三、刚发布的 v2.1.2 做了什么(本阶段成果)

这一版聚焦 **`/task` 子系统的参数解析与受保护工具授权**。5 个语义点,**新同事必须理解这些是当前契约,不要"修回"旧行为**:

### 1. dispatcher 是唯一自然语言解析路径(核心设计)
**文件**:`extensions/task/task-dispatcher.ts`

用户 `/task run <name> <自然语言>` 时,把自然语言翻译成结构化 `runtimeInput` 的是 **dispatcher**(一个 LLM 调用)。这是项目刻意的设计选择 -- **拒绝用字符串比对/正则猜测自然语言**(那是"老土的、治标不治本"的方式)。

边界(务必记住):
| 输入形式 | 谁解析 | 说明 |
|---|---|---|
| `{"url":"x"}` / `url=x` / `url: x` | `localRuntimeInput`(本地) | 确定性结构化语法,用户手写 |
| 裸值(`https://x`)、自然语言(`下载这个 https://x`) | **dispatcher**(LLM) | 唯一语义解析路径 |

### 2. dispatcher 不可用 -> 显式报错(不静默降级)
**关键**:当 dispatcher 的 model/auth 不可用时,**抛错**("dispatcher 模型不可用"),不再静默 `return undefined` 退化到本地解析或逐字段 UI 问。

**为什么**:旧行为(`return undefined` + `.catch(()=>undefined)`)会让 dispatcher 像是"可选的优化",于是不断有人想加本地正则兜底 -- 那正是项目要消灭的东西。显式报错让"dispatcher 没配好"成为一个清晰失败,逼用户配好或改用 field=value。

**错误类型区分**:
- 配置错误(无 model/auth) -> **抛错透传**
- `complete()` 运行时错误(网络/限流) -> `callDispatcher` 内部 catch 成 undefined(临时错误,fallback 合理)

### 3. 受保护工具授权缓存(per taskbook)
**文件**:`extensions/task/task.ts`(`protectedToolGrants`)

会话内授权过某 taskbook 用受保护工具(chrome_cdp / mcp)后,后续 `run_task` 不再弹 confirm。缓存 **key = `scope:name:sortedTools`**:
- `scope:name` 防同名 user/project taskbook 串扰
- `sortedTools` 让"taskbook 编辑后新增工具"必须重新确认

**注意**:这是 4 套独立权限门之一(见下),不是全局权限模型。

### 4. dispatcher `reasoningEffort = "medium"`
从 `minimal` -> `low` -> `medium`,因为它是唯一解析器,欠约束会导致解析不稳。

### 5. 本地 field=value 优先于 dispatcher/default
dispatcher 补全 required 后,显式 local(`page=2`)合回去压过 dispatcher 推断和默认值。

## 四、⚠️ 雷区:绝对不要做的事

这些是项目明确反向决定的,看到"建议"或老文档描述时**不要照做**:

1. **不要加本地自然语言解析捷径**(topN 正则、裸值抽取等)。曾被加过又撤掉(commit `1e7ad39`)。任何"对自然语言用正则"的想法都是治标不治本,正确做法是强化 dispatcher。

2. **不要让 dispatcher 静默降级**。`callDispatcher(...).catch(()=>undefined)` 这种写法已废弃。dispatcher 不可用就该报错。

3. **不要照抄 docs 里的旧代码片段**。v2.1.2 做了文档审阅,**6 份历史文档带 ⚠️ 废弃标注**,描述的是 v2.1.2 之前的旧行为(降级/fallback/return undefined)。它们开头都有醒目警告,**先读警告再读正文**。受影响文档:
   - `docs/handoff/2026-06-25-dispatcher-required-field-gate.md`
   - `docs/design/task-extension-followup-6.md`
   - `docs/design/task-extension-followup-4.md`
   - `docs/handoff/2026-06-24-subtask-review-fixes.md`(含可复制 stale 代码)
   - `docs/reports/2026-06-23-task-testing-report.md`(含"建议加 fallback 测试"的反向建议)
   - (一份 `docs/reports/2026-06-26-...-review.md` 已删除,因它把新设计当 bug 列)

4. **worker 子进程是全信任的,无沙箱**。`task-verify.ts:59` 直接 Node 跑 taskbook 的 verify 脚本;`task.ts` 的 `outputDir` 接受任意绝对路径。**taskbook 作者 ≡ taskbook 运行者**,加载 taskbook = 任意代码执行。这是已知设计取舍(单用户本地工具),不是 bug,但加任何"远程/不可信 taskbook"场景前必须先建沙箱。

5. **`pi` 不能单独更新,要更新整个 UGK**(AGENTS.md 明确)。

## 五、架构现状:权限是 4 套独立系统(未统一)

这是**最值得新同事关注的下一阶段课题**,也是上一阶段讨论过但没做的。

目前权限/确认分散在 4 处,**互不感知**:

| 系统 | 位置 | 管什么 |
|---|---|---|
| 危险 bash 门 | `extensions/index.ts:168-193` | `rm -rf`/`sudo`/`chmod 777`,交互 confirm/非交互硬拦 |
| chrome-cdp mode | `extensions/chrome-cdp/` (`/cdp ask\|on\|off`) | 进程内 mode,工具调用时查 |
| mcp mode | `extensions/mcp/` (`/mcp ask\|on\|off`) | server__tool 门 |
| task grant 缓存 | `extensions/task/task.ts:97` | per-taskbook 受保护工具授权 |

用户曾提出"做 Codex 风格的四模式(只读/询问/Approve for me/完全访问)"统一这些。**评估结论是:当前不能照搬**,因为 Codex 的"只读"靠 OS 级 sandbox(seatbelt/landlock),而 ugk/pi-core **没有 sandbox 层** -- 没有沙箱前提下做"只读模式"会变成假承诺(prompt 说只读,bash 照写)。

**真实待办**(按价值排序,供下一阶段参考):
1. 给 verify 脚本 / outputDir 加路径+命令围栏(堵 RCE 面) -- 比"统一权限模式"更有真安全收益
2. 把危险 bash 门**上抬命名**成全局 `ApprovalPolicy`(它已是事实 approval 层,只是没命名)
3. (远期)若要做完整四模式,必须先决定 worker 沙箱化方案

详见本会话历史或可重跑 ponytail + improve-codebase-architecture 评审。

## 六、开发约定(ponytail 风格)

项目改代码遵循 **ponytail** 原则(简洁、复用优先、最懒的正确版)。新同事开工前建议读:
- `C:\Users\29485\.agents\skills\ponytail\SKILL.md`(或 agent 运行时 `@ponytail`)

实操要点:
- **删多于加**:先看能不能删/复用,再考虑加。`resolveProtectedTaskTools` 就是因为重构后变死代码被删的(commit `1e7ad39`)。
- **commit 语义纯净**:逻辑改动在 fix commit,bump 版本/归档文档单独 commit。不要把代码改动塞进 `chore: bump version`。
- **测试是契约不是装饰**:现有测试很多是在**刻画**当前行为(包括边角)。改行为前先看测试在保护什么,改完跑全量。
- **加 `ponytail:` 注释**标记刻意的简化及其 ceiling(如 `// ponytail: 唯一保留的单字段确定性捷径`)。
- **危险操作前确认**(权限门已对 `rm -rf`/`sudo`/`chmod 777` 启用)。

**版本管理**:
- 版本号在 `package.json`,`bump` 是独立 `chore: bump version` commit
- 打 tag:`git tag v<version>`,推送:`git push origin main && git push origin v<version>`
- semver:bugfix -> patch,新 feature -> minor,破坏性 -> major
