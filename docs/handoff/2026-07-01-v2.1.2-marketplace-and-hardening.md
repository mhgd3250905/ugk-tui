## 1. UGK 是什么

**ugk**(读"乌克")是基于 [pi](https://github.com/earendil-works/pi)(pi-coding-agent)定制的终端 AI 编码 agent,装即用。

一句话定位:**把 pi 当作引擎,UGK 在上面叠了一层"任务系统 + 浏览器控制 + MCP 接入 + 定时任务 + 多 agent 协作 + 品牌化 TUI"的能力,并配套一个 task 分享市场。**

- 包名:`ugk-agent@2.1.2`,npm 全局安装,`ugk` 命令启动
- 引擎依赖:`@earendil-works/pi-*@0.79.4`(peer,跟着 pi 升级,**不能单独更新 pi,要更新整个 UGK**)
- 入口:`bin/ugk.js`(极薄包装,用 `-e` 注入 `extensions/index.ts`)
- 运行平台:Windows / macOS / Linux(bash 工具走 Git Bash,Windows 路径用正斜杠)

## 2. 当前版本能做什么(功能边界)

### 核心 agent 能力(继承自 pi)
- 多模型支持(DeepSeek 原生支持、各 provider 经 `/login` 配置)
- bash / edit / read / glob / grep 等内置工具
- plan 模式(只读探索)

## 3. 代码架构(改代码前必读)

### 目录结构

```
ugk-core/
├── bin/                    # CLI 入口(ugk.js + task-install.js + 各种 patch)
├── extensions/             # ← UGK 的全部逻辑都在这里(~11000 行 TS)
│   ├── index.ts            # 组合根:注册所有扩展/工具/命令/权限门
│   ├── task/               # task 系统(13 个文件,task.ts 2340 行是主)
│   ├── chrome-cdp/         # Chrome CDP 控制(7 文件)
│   ├── mcp/                # MCP 接入(7 文件)
│   ├── subagent*.ts        # 子代理委派
│   ├── ui-brand*.ts        # 品牌 UI / footer
│   ├── plan-mode*.ts       # 只读模式
│   ├── cron.ts             # 定时任务
│   ├── shared/             # 跨模块共享(binary/worker-lifecycle/language/autopilot/ui-language/settings)
│   └── ...
├── functions/              # Cloudflare Pages Functions(marketplace API,PR #27)
├── migrations/             # D1 迁移(marketplace,PR #27)
├── docs/task-share/        # 生成的静态市场站(由 scripts/build-task-share.mjs 生成,勿手改)
├── scripts/                # 构建脚本(build-task-share.mjs / smoke 测试)
├── tests/                  # 测试(见下)
├── skills/                 # 系统自带 skill(跟包走)
├── user-skills/            # 用户手动安装的 skill
├── prompts/ themes/        # 随包走的 prompts / 主题
└── docs/                   # 文档(design / handoff / reviews / reports)
```

### 关键架构约定(违反会被架构守卫测试拦截)

1. **task/ 不能 import chrome-cdp/mcp/plan-mode**(架构守卫:`tests/task-extension.test.ts:34` 用 `assert.doesNotMatch` 强制)。解耦靠 `shared/worker-lifecycle.ts` 中立层(DIP):chrome-cdp 注册 lifecycle 工厂,task-worker peek 出来用。

2. **shared/ 是共享层**,任何模块都能 import;task/ 只能 import shared/ 和 task/ 内部,不能反向依赖。

3. **权限门**(`extensions/index.ts:348`)拦截危险 bash(`rm -r*`/`sudo`/`chmod 777`),交互模式弹确认,非交互 fail-closed 直接拦。**autopilot(`/ugk-autopilot`)管不到权限门**——只放行可逆工具确认(CDP/MCP),危险命令永远归人。

4. **task 原子单元**:不可嵌套。`run_task` 工具的 parallel 模式必须用本工具,不能 subagent 包 run_task(subagent worker 会丢掉受保护工具授权)。

5. **生成产物勿手改**:`docs/task-share/**/*.html` 由 `scripts/build-task-share.mjs` 生成。改源头脚本后必须重新生成(`node scripts/build-task-share.mjs`),否则 16 个 HTML 的内联脚本不一致。

## 5. task 系统(核心,改 task 必读)

### contract.json 字段(全可选,向后兼容)

```json
{
  "outputDir": "<绝对路径或 runtime 默认>",
  "artifacts": [{ "name": "report.json", "type": "file", "required": true }],
  "runtimeInput": ["字段名1"],
  "runtimeInputMeta": {
    "字段名1": { "default": "...", "required": true, "allowedValues": ["a","b"] }
  },
  "requiredEnv": ["MIMO_API_KEY"],
  "requiredTools": ["chrome_cdp"],
  "requiredBinaries": ["yt-dlp", "ffmpeg"],
  "dispatcherModel": "...",
  "workerModel": "..."
}
```

### 三个依赖字段的分工(别混淆)

| 字段 | 管什么 | 校验方式 | 缺失行为 |
|---|---|---|---|
| `requiredEnv` | 环境变量(API key) | 读 process.env + Windows User env hydrate | `/task run` 弹窗问;`run_task` FAIL |
| `requiredTools` | **ugk 内部**受保护工具(chrome_cdp/MCP) | 走授权门 | 弹授权 confirm |
| `requiredBinaries` | **外部 CLI**(yt-dlp/ffmpeg) | `isBinaryAvailable` 查 PATH | `/task run` notify;`run_task` FAIL |

**关键区别**:`requiredTools` 是 ugk 内部的(走 CDP/MCP 授权),`requiredBinaries` 是系统命令(查 PATH)。**别把 yt-dlp 写进 requiredTools**。

### dispatcher 路由(v2.1.2 新逻辑)

- dispatcher 是 runtime input 的**唯一语义解析入口**(local field=value 不再 short-circuit)
- 新增 `allowedValues` 枚举校验:dispatcher 输出必须在允许值内,local 非法值不能覆盖 dispatcher 规范化结果
- dispatcher 部分成功时,**交互式 UI 预填有效值**(PR #26 审查修复),只问漏掉的字段
- 两条校验路径:`executeSubtask`(headless)+ `handleTaskRun`(交互式),顺序都是 env → binary,都在 worker 启动前

### progress 流式链路(易踩)

- 注入端:`subagent.ts` 的 `progressTextFromToolEvent`
- 接收端:`task-worker.ts` 的 dispatchWorker onUpdate 回调
- **content 流式文本无条件优先推送**(别改回 `!result?.messages?.length` 判定,多轮 progress 会丢)

## 8. 已知非阻塞债 / 待办

按优先级(高→低),**都不阻塞当前使用**,但接手后建议排期:

1. **Cloudflare R2 启用 + 部署**(运营):marketplace 生产部署的前置,非代码问题。启用 R2 → 建 bucket → 重跑 deploy。

2. **下载计数刷量防护**(marketplace):`recordDownload` 无去重无频率限制,`like/favorite/report` 都有 UNIQUE 唯独 download 没有。需计数语义/限流策略决策。

3. **CDP 端口随机化**(安全加固):launcher 用固定 9222。改 `--remote-debugging-port=0` + 显式 `--remote-debugging-address=127.0.0.1` 可消除固定端口可预测性。

4. **integration 测试缺口**:task/chrome-cdp/subagent 缺端到端 integration 层(目前只有 MCP 有)。

5. **requiredBinaries 不校验版本**(task):有意为之(版本校验是深坑),只验"存在"。将来需要版本校验是独立大需求。

6. **executeSubtask headless FAIL 路径无单独测试**(task):逻辑极简,requiredEnv/requiredBinaries 各有间接覆盖。

7. **docs 卫生**:部分 handoff 文档仍指向已删的 `driver-session.ts`(改名 `driver-view.ts`),仅文档残留非代码债。

## 10. 工作风格(本项目一直遵循的)

- **ponytail full**:最短可用 diff、根因修复、非平凡逻辑留可跑的 check、`ponytail:` 注释标刻意简化
- **对抗性审查**:复杂改动派 sub agent 审(用 ponytail skill),然后**逐条核实**——真问题修、误报/over-engineering 不采纳。**sub agent 会标错代码位置/报过时结论,必须自己读源码复核**
- **测试抓回归**:每个修复都改坏验证(临时改回 buggy 确认测试转红)
- **真机验证不可省**:安全/进程相关,单元测试给假绿是常态(本轮 Chrome teardown 的教训:集成测试用 cmd 模拟进程树,但 Chrome stub-exit 行为不同,测错模型)
