## 1. 本会话做了什么(从零到上线)

### 1.1 核心功能:TUI 内 `/task publish` 上传到市场

用户在 TUI 跑出 landed task → `/task publish` → 上传到 task 分享市场,无需切浏览器。

**CLI auth 链路**(经两轮对抗式审查 + 多次真机验证才稳定):

```
TUI 生成 challenge → POST /api/cli/auth/start(记 pending)
  → 浏览器开 /cli-auth?c=<challenge>
  → 纯跳转 /api/auth/github?cli_challenge=<challenge>
  → githubLogin 服务端绑定 state↔challenge(写 pending.state)
  → GitHub OAuth → callback
  → callback 用已验证的 state 反查 challenge(不读 JS cookie)
  → 重定向 /cli-auth?cli=confirm&state=<state>(用户确认页)
  → 用户点 Authorize → POST /api/cli/auth/confirm(session cookie 鉴权)
  → createCliToken(DELETE...RETURNING 原子 claim)签 cli_token
  → TUI 轮询 /api/cli/auth/poll 拿 token → 存 ~/.pi/agent/task-share.json
  → 后续上传带 Authorization: Bearer <token>
```

### 1.2 市场前端(合并 PR #29)

## 4. 本会话踩过的坑(接手必读,真机才能暴露)

### 4.1 Workers Illegal invocation(crypto.randomUUID)
`createCliToken` 原写法 `(deps.randomUUID ?? crypto.randomUUID)()` —— `crypto.randomUUID` 当裸函数引用调用,丢失 `this` 绑定。**Node 宽容不报错(单测全过),Workers 严格校验抛 `Illegal invocation`**(前端看到 Error 1101)。
- 根因靠 debugLog 才定位(token_exchange/user_fetch 都成功,崩在 createCliToken)
- 修复:`deps.randomUUID ? deps.randomUUID() : crypto.randomUUID()` 直接在 `crypto` 上调用
- **教训**:任何 `obj.method` 脱离对象调用,在 Workers 都可能 Illegal invocation。Node 单测抓不到这类。

### 4.2 Pages 分支部署 vs 主域名
`wrangler pages deploy --branch <feature>` 生成别名域名,但**主域名仍指旧 main**。curl 主域名发现新 Functions 没生效(404/旧行为)。必须 `--branch main` 部署才更新主域名。

### 4.3 `/cli-auth` URL 规范化
Pages 对 `/cli-auth`(无尾斜杠)返回 308 到 `/cli-auth/`(保留 query)。curl 测试要加 `-L`。浏览器自动跟随,不影响真实用户。

### 4.4 push 前终审发现的 4 个阻断项(都已修)
| 问题 | 级别 | 修复 |
|---|---|---|
| debugLog 明文写 GitHub access_token + user PII | CRITICAL | 删敏感字段,只记 status+login |
| OAuth 登录型 CSRF(challenge 注入受害者 OAuth 静默签 token) | HIGH | callback→确认页→confirm 端点(session 鉴权) |
| 双 tab 孤儿 token(TOCTOU,SELECT/INSERT/DELETE 非原子) | HIGH | createCliToken 用 DELETE...RETURNING 原子化 |
| 90 天过期后客户端无重授权(opaque 报错) | HIGH | handleTaskPublish 捕获 invalid_token 清 token |

```bash
# 1. 改 package.json version
```
