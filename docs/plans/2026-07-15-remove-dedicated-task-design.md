# 移除专用 task 设计

## 决策

UGK 不再区分“普通 task”和“专用 task”。产品定位已经转为通过 MCP 执行本机已安装、已验收的 task，因此隐藏一部分 task 再要求 Agent 读取额外索引，只会增加匹配失败、工具冲突和理解成本。

## 行为

- `buildTaskbookPrompt` 始终列出全部 task，不再生成或引用 `_dedicated-index.md`。
- `/task` 的列表、选择器和详情菜单不再显示锁图标，也不再提供“设为专用/取消专用”。
- `dedicated` 不再是特殊标签。旧 taskbook 即使仍带有该字符串，也与其他普通标签一样，不会影响可见性、匹配或执行。
- MCP 和旧 TUI 使用同一份完整 task 清单，避免两套发现规则再次分叉。

## 兼容与清理

不增加运行时迁移器：旧 JSON 仍能正常读取，`tags` 的其他用途不变。本机现有 taskbook 中的 `dedicated` 标签直接移除；其他用户升级后即使标签暂时残留，行为也已等同普通 task。旧索引文件不再被读取，可安全删除。

## 验证

回归测试覆盖：带旧 `dedicated` 标签的 task 会出现在启动提示和选择器中；菜单没有专用切换入口；`run_task` 仍可正常执行。最后运行完整单元测试与 MCP 集成测试。
