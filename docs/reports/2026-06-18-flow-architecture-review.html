index.ts 退化为"命令→调度"薄层;orchestrator 一个接口(prove/run/review/accept/reject)背后藏住所有 gate、store 写入顺序、repair 重试。测试只需对着 orchestrator 接口锤,不再 mock 5 个 store。

Problem

prove/run/review 三条生命周期的 gate 判定、store 写入顺序、repair 单次重试全写在 index.ts 的命令 handler 和 .then() 链里。改一条 gate 要在 1315 行里穿行,locality 为零。

Solution

抽出 orchestrator.ts:接口为 prove/run/review/accept/reject 五个动词,实现吸收全部 gate + store wiring + repair 重试。index.ts 只剩命令路由与 UI/事件。

Wins
  • locality:gate 逻辑集中到 1 个 module
  • leverage:5 个动词接口,N 个命令 handler 复用
  • interface 是测试面:不再 mock 5 个 store
  • index.ts 从 1315 → ~300 行
  • repair 重试/accepted backfill 各自归位
  • 命令路由与领域规则物理隔离

Top recommendation

先做 ①(Task Lifecycle Orchestrator)

候选②③⑤是低风险清理(机械重构 + 删 dead code),可以并行顺手做。但真正阻止 flow 模块变成屎山的是 ①: 只要 index.ts 还把 5 道 gate + store wiring + repair 重试 + 3 条生命周期捏在一个闭包里,后面每加一个状态、每加一道 gate,都会让 god module 再胖一圈。 ① 落地后,② 的 IO 收敛和 ④ 的 DriverViewPort 自然有了归处(orchestrator 持 viewPort,store 被 orchestrator 统一调)。

建议顺序: ⑤(清场)→ ③(状态谓词,给①铺路)→ ①(主菜)→ ②(IO 收敛)→ ④(UI seam)。 ①③ 是同一条 deepening 的两步:先把可运行状态收敛成领域谓词,再把整个生命周期(含该谓词)收进 orchestrator。
